# 프로덕션이라면 — 의도적 미구현과 그 근거

> 설계 문서 §5-2 의 실파일화. 이 프로젝트는 **포트폴리오 스코프**로, "무엇을 안 했는가"와
> "왜 안 했는가"를 명시하는 것이 "무엇을 했는가"만큼 중요하다고 본다. 아래 항목은 *못 한 것*이
> 아니라 **스코프 밖으로 의식적으로 밀어낸 것**이며, 각 항목에 프로덕션 전환 시의 실제 경로를 적는다.
>
> 판단 기준: **이 프로젝트가 증명하려는 것은 "멀티 게스트 실시간 미디어의 정확성·복구·회귀 차단"이다.**
> 그 논지에 기여하지 않는 인프라·플랫폼 배관은, 업계 표준 해법이 확립돼 있는 한 스코프에서 제외했다.

## 1. 실인증 (real identity)

**현재**: `POST /auth/session` 이 `{nickname}` 만 받아 user 를 만들고 세션 JWT 를 발급한다.
role 은 서버가 방 상태(DB)로 결정하므로(자기선언 불가) **권한 경계 자체는 견고**하지만, "이 닉네임이
누구인지"에 대한 신원 보증은 없다.

**프로덕션이라면**: OAuth(소셜) 또는 이메일 매직링크 + 세션 회전. 방 소유권·차단 목록을 닉네임이
아니라 안정적 user id 에 묶고, host 권한을 계정 단위로 승계 가능하게 한다.

**왜 지금 안 했나**: 인증은 해법이 확립된 배관이고, 이 프로젝트의 논지(미디어 정확성)에 기여하지
않는다. 대신 **인증과 무관하게 성립해야 하는 것** — "role 은 절대 클라이언트가 자기선언할 수 없다"
(`/rooms/:id/token` 이 `guest`/`speaker` 자기선언을 403) — 은 지금 구조로 강제된다. 실인증을 얹어도
이 경계는 바뀌지 않는다.

## 2. TURN 자가운영

**현재**: 프로덕션 미디어는 LiveKit Cloud(관리형 SFU + TURN + TLS)에 위임한다. 로컬은 OSS
LiveKit 컨테이너를 self-host 하되, TURN 없이 host-reachable ICE 후보(`node_ip`)로만 동작한다
(대칭 NAT·엄격 방화벽 뒤 클라이언트는 로컬에서 연결 실패 가능 — 알려진 개발 한계).

**프로덕션이라면**: 두 갈래.
- (a) **Cloud 유지** — TURN 을 LiveKit Cloud 에 맡기고 앱은 미디어 인프라를 소유하지 않는다.
  운영 부담 최소, 대신 대역폭 과금이 트래픽에 비례.
- (b) **자가운영** — coturn 을 공인 IP 로 띄우고 LiveKit 에 물린다. 대규모에서 relay 대역폭
  단가를 통제할 수 있으나, TURN 은 상태·대역폭·보안(자격증명 회전, 오남용 차단)이 만만치 않은
  독립 운영 대상이 된다.

**왜 지금 안 했나**: TURN 자가운영의 난도는 "미디어가 어려워서"가 아니라 "네트워크 인프라 운영이라서"다
— 이 프로젝트가 증명하려는 역량과 결이 다르다. 아키텍처는 self-host/Cloud 를 **env 스위치**로 이미
분리해 두어(코드 변경 0), 어느 쪽으로도 전환 가능한 상태를 유지하는 것으로 갈음한다.

## 3. 단일 방 확장 한계와 티어링 전략

이 시스템은 **한 방의 정확성**에 집중한다. 확장은 이미 두 축으로 설계돼 있고, 남은 축은 명시적 과제다.

**이미 구현된 티어링 — 시청자 축**:
- **모드 A (WebRTC subscribe-only)**: 시청자가 같은 LiveKit room 에 구독 전용으로 입장. 저지연,
  ~수천 규모. SFU 팬아웃에 의존.
- **모드 B (Egress → HLS + CDN)**: Room Composite → HLS 세그먼트 직접 출력 → CDN 캐싱.
  지연은 늘지만 시청자 수는 CDN 한계까지(사실상 무제한). `viewer_mode` 는 방 생성 시 고정.
- egress 실패 시 자동으로 모드 A 폴백 + host 알림 — 티어 경계에서도 시청은 끊기지 않는다.

**미구현 — 남은 확장 축과 경로**:
- **API 수평 확장**: 현재 단일 Fastify 인스턴스. 상태는 이미 Postgres/Redis 로 외재화돼 있어
  스테이트리스 다중화가 가능하나(webhook 멱등·Redis 락 전제), 로드밸런서·세션 어피니티·webhook
  팬아웃을 실측 검증하진 않았다.
- **방 개수(수평) 확장**: 방 스위퍼(host 유예 종료)·presence 가 단일 프로세스 타이머 가정. 다중
  인스턴스에서는 리더 선출 또는 분산 스케줄러로 옮겨야 중복 종료를 막는다(현재도 SETNX 락으로
  레이스는 방어하나, 스케줄러 자체의 분산화는 과제).
- **방 하나의 인원 상한**: guest 8 / speaker 20 은 **제품 결정**(운영 가능한 무대 규모)이지 기술
  상한이 아니다. 더 키우면 host UI 의 타일 페이지네이션·오디오 도달 리포트 비용(참가자² 대조)이
  먼저 병목이 된다 — 그때는 리포트를 샘플링/증분화하거나 서버측 구독 추적으로 옮긴다.
- **host 승계**: 현재 host 이탈은 유예 후 방 종료. 승계(다음 권한자에게 이양)는 명시적 백로그.

**왜 이 선을 그었나**: "한 방이 100% 정확하고 자동 복구되는가"를 증명하지 못하면 수평 확장은
의미가 없다. 정확성 불변식(오디오 전수 도달)과 그 회귀 차단을 먼저 고정하고, 확장은 **상태 외재화 +
멱등 + 락**이라는 다중화 전제까지만 갖춘 뒤 실측은 과제로 남긴다. 조기 수평 확장은 증명되지 않은
정확성을 더 넓게 복제할 뿐이다.

## 4. 그 밖의 의식적 스코프 컷

- **관측 → 경보**: `/metrics`(prom-client)로 오디오 누락·egress 전이·webhook 오류를 노출하지만,
  Alertmanager 규칙·온콜 연동은 없다. 지표는 준비됐고 경보 배선만 남긴 상태.
- **시크릿 관리**: env + 부트 fail-fast(프로덕션 약한 시크릿·CORS `*` 거부)까지. Vault/KMS 연동,
  자동 회전은 미구현.
- **데이터 수명주기**: 데모 모드의 방 수명·데이터 보존 가드는 있으나, 정식 보존정책·GDPR류
  삭제 요청 처리·감사로그 장기 보관은 스코프 밖.
